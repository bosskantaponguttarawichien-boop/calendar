# รายงานการปรับปรุงโครงสร้างข้อมูล (Data Optimization)

จากการวิเคราะห์ข้อมูลที่เก็บใน Firestore ของแอปพลิเคชัน Calendar พบว่ามีข้อมูลบางส่วนที่ซ้ำซ้อนและสามารถปรับปรุงเพื่อลดพื้นที่จัดเก็บ รวมถึงทำให้การจัดการข้อมูลง่ายขึ้นในอนาคต ดังนี้:

## 1. ฟิลด์ที่ซ้ำซ้อน (Redundant Fields)

| ฟิลด์ | สถานะ | เหตุผล |
| :--- | :--- | :--- |
| `category` | **ไม่จำเป็น** | มีค่าเดียวกับ `shiftId` แทบทุกกรณี สามารถดึงข้อมูลจาก `shiftId` มาแสดงผลแทนได้ |
| `icon` | **ไม่จำเป็น** | รูปไอคอนผูกกับประเภทของเวรอยู่แล้ว สามารถทำ Icon Mapping ใน Frontend ได้ |
| `title` | **อาจไม่จำเป็น** | ถ้าเป็นเวรมาตรฐาน (เช่น "เช้า", "บ่าย") สามารถดึง Label มาแสดงตาม `shiftId` ได้เลย |

## 2. การจัดการสี (Color Management)

> [!IMPORTANT]
> **ปัญหาปัจจุบัน:** มีการเก็บค่า Color Code (เช่น `#00AB84`) ลงในทุก Record

*   **ข้อเสีย:** หากต้องการเปลี่ยนสีของ "เวรเช้า" ในอนาคต จะต้องแก้ไขข้อมูลทุก Record ใน Database
*   **แนวทางแก้ไข:** เก็บเฉพาะ `shiftId` แล้วให้ Frontend ไปดึงสีจากไฟล์ Constants (เช่น `CATEGORIES`) มาแสดงผลแทน ช่วยให้การแก้ไขสีในภายหลังทำได้ที่จุดเดียว

## 3. ช่วงเวลา (Start & End Time)

*   **ข้อสังเกต:** สำหรับกิจกรรมที่เป็น "ทั้งวัน" (All-day) ค่า `start` และ [end](file:///c:/Users/boss_/OneDrive/%E0%B9%80%E0%B8%AD%E0%B8%81%E0%B8%AA%E0%B8%B2%E0%B8%A3/GitHub/calendar/src/screens/home-screen/hooks/useCalendarController.ts#14-236) มักจะตรงกันเป๊ะ
*   **แนวทางแก้ไข:** หากเป็นกิจกรรมที่ไม่ระบุเวลาเจาะจง อาจเลือกเก็บเป็น String ของวันที่ เช่น `date: "2027-06-22"` แทนการใช้ Timestamp เต็มรูปแบบ เพื่อความง่ายในการ Query และประหยัดพื้นที่

## 4. การแยกตาราง Shifts (Normalized Shifts Table)

> [!IMPORTANT]
> **สถานะปัจจุบัน:** กำลังดำเนินการแยกโครงสร้าง (Implementing) โดยใช้ Hooks แยกต่างหาก (`useShiftService`, `useShiftController`) เพื่อจัดการข้อมูลเวรโดยเฉพาะ

> [!TIP]
> **ข้อเสนอแนะเพิ่มเติม:** แทนที่จะเก็บข้อมูลเวรพิเศษลงในทุก Event แนะนำให้แยกไปเก็บใน Collection เฉพาะ (เช่น `/users/{uid}/shifts/`) แล้วอ้างอิงผ่าน `shiftId` เท่านั้น

*   **ความง่ายในการจัดการ (Maintainability):** หากต้องการแก้ไขชื่อเวรหรือเปลี่ยนสี จะทำได้ที่จุดเดียวและส่งผลต่อทุก Event ทั้งในอดีตและอนาคต
*   **ประสิทธิภาพ (Performance):** โค้ดใน Frontend (เช่น `EventModal`) ไม่ต้องทำการค้นหา (scan) ข้อมูลจาก events ทั้งหมดมาเพื่อสร้างรายการเวรพิเศษ แต่สามารถดึงจากตาราง Shifts ได้โดยตรง
*   **ความสม่ำเสมอของข้อมูล (Data Consistency):** ป้องกันการเก็บสีหรือชื่อเวรที่ผิดเพี้ยนไปในแต่ละวัน

### โครงสร้างที่แนะนำเมื่อมีการแยกตาราง:

**`/shifts/{shiftId}`** (ตารางแม่):
```json
{
  "id": "shift_001",
  "title": "Evening Shift",
  "color": "#FF5733",
  "icon": "Moon",
  "startTime": "16:00",
  "endTime": "00:00"
}
```

**`/events/{eventId}`** (ตารางลูก):
```json
{
  "userId": "U13...",
  "date": "2024-03-24",
  "shiftId": "shift_001"
}
```
